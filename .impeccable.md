## Design Context

### Users
- **Who they are**: Platform engineers, developer experience leads, and repository maintainers.
- **Their context**: Monitoring and tuning linting/enforcement rules for AI agents and human developers in CI or local git hooks.
- **Job to be done**: Identify which rules are "too noisy" (advisory pressure), which rules are crashing (runtime errors), and which rules are inconsistent (decision variance) to calibrate rule configurations and keep coding loops fast and high-quality.

### Brand Personality
- **Voice/Tone**: Clinical, industrial, high-signal, unimpressed.
- **3-word personality**: Precise, Dense, Utilitarian.
- **Emotional goals**: Focus, confidence in metrics, clarity in rule diagnostics.

### Aesthetic Direction
- **Visual tone**: Obsessively organized developer dashboard. Structured dark theme, fine-lined borders, high-contrast status colors.
- **Fonts**: Pairing `Inter` (sans-serif) for body text and labels to keep reading clean, with `JetBrains Mono` for rule IDs, numbers, and console elements to retain mechanical telemetry feel.
- **Anti-references**: Neon gradients, glossy glassmorphism, rounded card grids, uppercase typography, side-stripe card borders.
- **Theme**: Dark mode.

### Design Principles
1. **Clinical Data Density**: Prioritize visual alignment and clean typographic hierarchy over decorative accents. Every pixel of padding must establish structural rhythm.
2. **Action-oriented Telemetry**: Group and present numbers in a way that suggests clear triage decisions (e.g., "Tune policy", "Inspect evidence").
3. **Focus on Semantics**: Interactive elements must feature precise state changes (e.g., subtle row highlights with select indicators instead of aggressive color changes).
4. **Purposeful Transition**: Decelerate UI elements smoothly using quartic/quintic curves, preventing layout shifts and respects `prefers-reduced-motion`.
